The one idea: PyTorch is the glue. Tensors carry the data, autograd computes the gradients, the optimizer applies them, and the dispatcher sends every operation to a fast GPU kernel. Learn how the pieces snap together and you can read almost any training script.

1 The tensor, now with gradients

In plain words: a PyTorch tensor is the flat-buffer-plus-metadata array from #2 — with one addition: it can remember every operation done to it, so gradients can flow back.

It's the same tensor from explainer #2 (storage + shape + strides + dtype). The new trick: set requires_grad=True and the tensor starts recording what happens to it.

PyTorch a tensor that tracks gradients
import torch
w = torch.randn(3, requires_grad=True)   # a weight to learn
y = (w * 2).sum()                      # do some math
y.backward()                            # fill w.grad via autograd
w.grad                                  # tensor([2., 2., 2.])

Every model parameter is just a tensor with requires_grad=True. That single flag is what turns a plain array into something trainable.

2 The autograd tape

In plain words: as you run the forward pass, PyTorch quietly records each operation on a "tape." Calling .backward() replays that tape in reverse to compute gradients — exactly the tiny autograd from #4, at scale.

This is #4's Value-remembers-its-parents idea, built into the framework. When you operate on tensors that need gradients, PyTorch:

  • Runs the op and records a backward node on the result (its grad_fn, e.g. MmBackward for a matmul).
  • Builds these nodes into a graph as the code runs — "define-by-run." The graph can change every iteration (loops, ifs all work), and it's rebuilt fresh each step.
  • On loss.backward(), walks that graph backward from the loss, filling each parameter's .grad.
One gotcha worth knowing: PyTorch accumulates into .grad rather than overwriting — handy for some tricks, but it's why every training loop calls zero_grad() first. Forget it and this step's gradients pile on top of last step's.

3 nn.Module: stacking the layers

In plain words: an nn.Module is a tidy container for layers and their weights. You define what the forward pass does; PyTorch tracks all the parameters for you.

A model is a stack of layers, and each layer is mostly a matmul (#3: y = W·x) plus an activation. nn.Module is the standard way to bundle them: declare the layers, write the forward, and every weight inside is automatically a tracked parameter.

PyTorch a tiny two-layer network
import torch.nn as nn

class Net(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc1 = nn.Linear(784, 128)   # a matmul + bias
        self.fc2 = nn.Linear(128, 10)

    def forward(self, x):
        x = torch.relu(self.fc1(x))    # matmul → activation
        return self.fc2(x)             # → logits

Calling model(x) runs forward (and records the autograd tape along the way). model.parameters() hands you every weight — which is exactly what the optimizer needs next.

4 The optimizer: gradient descent, wrapped

In plain words: the optimizer holds all the model's weights and, on command, nudges each one downhill using the gradients autograd computed. It's #4's gradient descent in a box.

You hand the optimizer every parameter once, up front. Then each step it applies the update from explainer #4 — weight −= learning_rate × grad — to all of them at once:

PyTorch create an optimizer
opt = torch.optim.Adam(model.parameters(), lr=1e-3)
# opt.step()  applies weight -= lr * grad to every parameter
# opt.zero_grad()  clears grads before the next backward

Adam and friends are just smarter versions of gradient descent — they adapt the step size per weight — but the idea is identical: use the gradient to move toward lower loss.

6 Onto the GPU

In plain words: one method — .to('cuda') — moves a tensor or model onto the GPU. After that, every operation automatically runs as a GPU kernel from explainers #1–3.

A tensor lives on a device — CPU or GPU. Move the model and the data to the GPU and you're done:

PyTorch move to the GPU
device = "cuda" if torch.cuda.is_available() else "cpu"
model = model.to(device)
x = x.to(device)
out = model(x)        # now every op runs on the GPU

Here's the link back to the whole first half of the series: when you write a @ b, PyTorch's dispatcher looks at where the tensors live and routes the call to the right kernel — on a CUDA tensor, that's cuBLAS (explainer #3's tuned matmul), running on the SMs and tensor cores from #1. You write one line of Python; under it is everything from explainers #1–3.

7 Eager vs. torch.compile

In plain words: by default PyTorch runs line by line (easy to debug). Wrap your model in torch.compile and it captures the whole graph and fuses it into faster kernels.

PyTorch's default is eager mode — "define-by-run," where each line executes immediately. That's why it's so pleasant to debug: you can print a tensor mid-model, set a breakpoint, use normal Python control flow. The cost is that the GPU does one op at a time, with Python overhead between them.

torch.compile (one line: model = torch.compile(model)) captures your model into a graph the first time it runs, then fuses operations into optimized kernels and reuses them — often a sizable speedup, with little code change. The trade-off is the classic one from the series: eager is flexible and easy; compiled is faster but more rigid.

The mental model: eager = an interpreter (run now, easy to inspect); compile = a JIT compiler (capture, optimize, then run fast). Start in eager, reach for compile when you need the speed.

✓ The whole picture, in one breath

PyTorch is the four pieces of this series, assembled. A tensor (#2) with requires_grad records an autograd tape as you run; an nn.Module stacks layers (each a matmul, #3) into a model whose weights are tracked parameters; an optimizer applies gradient descent (#4) to those weights. The training loop — zero_grad, forward, loss, backward(), step() — is #4's five steps in real code. Call .to('cuda') and the dispatcher sends every op (like a @ b) to a GPU kernel such as cuBLAS (#1–3). It runs eager by default for easy debugging, and torch.compile fuses the graph when you need speed.

tensor + autograd→nn.Module→optimizer→training loop→.to(cuda) → cuBLAS
