Matmul: the operation that runs the world

Almost all of an AI model's compute is one humble operation — matrix multiply. The math is trivial. Making it fast is the whole game, and it turns out to be a memory problem, not a math one.

  • why the naive version is slow — and what “memory-bound” means
  • how tiling and tensor cores make it fast
  • why linear layers, convolutions, and attention are all matmul

▶ Start here: the trivial loop that eats every GPU

In plain words: matrix multiply is three nested loops you could write in a minute — and it's where nearly all of an AI model's time goes. The hard part isn't the math; it's feeding the math units fast enough.

You've met this idea twice already: explainer #1 (§10) and #2 (§6) both said AI is mostly matmul. Here's the operation itself — the textbook triple loop:

Matrix multiply C = A · B
for i in range(N):
  for j in range(N):
    for k in range(N):
      C[i][j] += A[i][k] * B[k][j]

That's the whole operation: every cell of C is a row of A dotted with a column of B. Nothing clever. Yet training a large model can spend 90%+ of its time right here. So the interesting question isn't "what does it compute" — it's:

The math is dead simple — so why is fast matmul so hard?

The answer, in one line: a GPU can do the multiplies far faster than it can fetch the numbers to multiply. Fast matmul is the art of reusing data so the math units never starve. Everything below is that one idea.

1 What matmul computes (and how much)

In plain words: each output cell is one dot product. There are N² cells, each costing ~2N operations — so the work grows as N³.

For an N×N times N×N multiply, every output cell C[i][j] is a dot product down a row of A and a column of B: N multiplies and N adds. So:

  • N² output cells, each ≈ 2N FLOPs (a multiply + an add, N times) → 2N³ total FLOPs. Double N and the work goes up 8×.
  • Each output cell is independent — exactly the parallelism GPUs love (one thread per cell, from explainer #1 §10).

The math couldn't be simpler or more parallel. The trouble is entirely in where the numbers live.

2 The naive kernel re-reads everything

In plain words: the obvious one-thread-per-cell kernel fetches every input from slow memory over and over — the same numbers, thousands of times.

The natural GPU version is explainer #1's pattern: one thread per output cell, looping over k:

Naive matmul kernel one thread per output cell
int row = blockIdx.y*blockDim.y + threadIdx.y;
int col = blockIdx.x*blockDim.x + threadIdx.x;
float acc = 0;
for (int k = 0; k < N; k++)
    acc += A[row*N + k] * B[k*N + col];   // both reads hit slow global memory
C[row*N + col] = acc;

It's correct, and it works. But look at the reads. Every thread pulls a full row of A and a full column of B straight from global memory (the slow, far tier from #1, §3). And the thread next door, computing the cell beside it, re-reads almost the same row of A all over again.

Worked example — N = 1024:
• Math: 2·N³ ≈ 2.1 billion FLOPs.
• Memory (naive): each cell reads 2N numbers; over N² cells that's 2·N³ ≈ 2.1 billion reads — meaning every element of A and B is fetched ~1,024 times.
• That's only ~0.25 FLOPs per byte (4-byte floats). An A100 needs roughly 12+ FLOPs/byte to keep its math units fed — so the naive kernel leaves the cores almost entirely idle, waiting on memory.

3 Memory-bound vs compute-bound (the roofline)

In plain words: divide a kernel's math by the bytes it moves — that ratio (arithmetic intensity) decides whether the chip is limited by its math units or by memory bandwidth.

Arithmetic intensity = FLOPs ÷ bytes moved. Every chip has a break-even point (its "ridge"): below it you're memory-bound — cores idle, starved for data; above it you're compute-bound — finally limited by raw math, which is what you want. Naive matmul sits at 0.25, deep in the memory-bound zone.

The fix is to reuse each number more times per trip to memory. Drag the reuse factor and watch arithmetic intensity climb across the ridge:

4 Big idea #1: shared-memory tiling

In plain words: load a small square tile of A and B into the SM's fast shared memory once, and let every thread in the tile reuse it — instead of each thread fetching from slow memory itself.

Here's the reuse from §3, made real, using the shared memory scratchpad from explainer #1 (§3). Instead of every thread reading from global memory, the block cooperates:

  • Threads load one T×T tile of A and one of B from global memory into shared memory (fast, on-chip) — with coalesced, neighbour-friendly reads (#1, §8).
  • Every thread then does its multiply-adds reading from shared memory, reusing each loaded value T times.
  • Slide to the next tile along k, accumulate, repeat.

Net effect: each element comes from slow global memory once per tile instead of N times. Global reads drop by roughly a factor of T, arithmetic intensity rises ~T×, and the kernel crosses the ridge into compute-bound — exactly the slider in §3.

The takeaway: tiling doesn't change the math by one FLOP — it changes how often you touch memory. That's the whole job: keep hot data close and reuse it, so the math units stay fed. (This is also why the §1 memory hierarchy exists.)

5 Big idea #2: tensor cores

In plain words: tensor cores are hardware that multiplies a whole small matrix in a single instruction — purpose-built for exactly this operation.

Tiling feeds the math units; tensor cores make the math units themselves enormously faster at this one operation. Instead of doing one multiply-add at a time, a tensor core multiplies a small matrix tile (e.g. 16×16) and accumulates it in a single operation — dozens of multiply-adds at once.

The catch (and the link back): they run fastest at low precision — fp16 / bf16 from explainer #2 (§4). That's why those dtypes matter so much, and why a GPU's fp16 tensor-core throughput is ~10× its fp32 number (#1, §1 — FLOPS depends on precision). Tensor cores are the same specialization that, taken to the extreme, becomes a TPU (#1, §11).

Two levers, one goal: tiling keeps the data flowing; tensor cores crunch it in bulk. Together they turn matmul from memory-starved to near the hardware's peak.

6 Why this one operation is worth a whole explainer

In plain words: the big operations in a neural network — linear layers, convolutions, attention — are all matrix multiplies in disguise. Speed up GEMM and you speed up all of AI.

GEMM (general matrix multiply) is the workhorse the whole field is built on:

  • Linear / fully-connected layers — literally y = W·x. A matmul.
  • Convolutions — unrolled into a matmul (im2col) or run as an "implicit GEMM."
  • Attention (the engine of transformers, coming in #6) — a stack of matmuls: Q·Kᵀ, then ·V.

So this isn't one operation among many — it's the operation. Every hour spent optimizing GEMM pays off across every model, which is exactly why GPUs, tensor cores, and TPUs are all shaped around it.

7 In practice: don't write your own

In plain words: you almost never hand-write matmul. You call a tuned library that's already squeezed the hardware dry.

Understanding tiling and tensor cores is essential for reasoning about performance — but in real code you call cuBLAS (or cuDNN, CUTLASS). These are hand-tuned to each GPU, juggling tile sizes, bank conflicts, vectorized loads, and warp-level scheduling. Simon Boehm's worklog shows it takes around a dozen escalating optimizations for a careful hand-written kernel just to approach cuBLAS — and PyTorch already calls these libraries for you under C = A @ B.

The honest takeaway: know how matmul gets fast so you can spot when you're memory-bound or using the wrong precision — but reach for the library to actually run it.
