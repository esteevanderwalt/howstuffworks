▶ Start here: from a loop to a million threads

In plain words: a GPU runs your loop by doing every step at the same time, instead of one after another.

You've written this a hundred times. Add two arrays element by element:

On a CPU the way you'd write it today
for (int i = 0; i < N; i++)
    C[i] = A[i] + B[i];

For N = 1,000,000 this runs one iteration at a time — a million trips through the loop, in order. Even at billions of instructions per second, C[i] can't start until C[i-1] is done.

Now look hard at the loop body. Iteration 5 never reads anything iteration 4 wrote. Every iteration is completely independent. They only run in sequence because a single CPU core executes a single stream of instructions. The ordering is a fact about the hardware — not about the problem.

The GPU's bet: if the iterations don't depend on each other, why do them in order at all? Spawn a million tiny workers at once, give each one a single value of i, and let them all run simultaneously.

One worker is a thread. The loop body becomes the kernel — and notice the for vanishes entirely:

On a GPU the kernel — what ONE thread runs
int i = blockIdx.x*blockDim.x + threadIdx.x;  // which i am I?
C[i] = A[i] + B[i];                          // the old loop body, once

There is no loop. The GPU stamps out a million copies of these two lines, each copy carrying a different i. That first line is just every worker asking "which iteration am I responsible for?" — the interactive in §5 shows exactly how the math hands each thread a unique i.

“A million threads?!” — first, drop your CPU instincts. If you already know threads from CPUs (pthreads, Java threads, Python threading), that mental model will fight you here. A GPU thread is a different animal:

So when we say a million threads, picture a million tiny disposable workers — not a million operating-system threads. That difference is the GPU.

That single move — turn the loop body into one thread per iteration — is the entire idea. Everything else on this page exists to answer one engineering question:

How do you actually run a million threads on a chip that doesn't have a million cores?

A million won't fit at once.
→ Group threads into blocks, hand each block to one SM, queue the rest. §2, §4
True independence costs silicon.
→ Run threads in lockstep bundles of 32 (warps) that share one instruction decoder. §6
Threads keep waiting on memory.
→ Keep thousands resident and swap between them for free, so the math units never idle. §6, §7
Memory is far and slow.
→ A hierarchy from tiny-fast to huge-slow, plus code that keeps hot data close. §3

Read the rest as answers to that question, not a list of terms to memorize.

1 CPU vs GPU: two different goals

In plain words: a CPU is a few fast workers; a GPU is thousands of slower ones. For one quick job the CPU wins; for a mountain of identical jobs, thousands of hands win.

The core difference isn't speed — it's what they optimize for. A CPU is built to finish one task as fast as possible (low latency). A GPU is built to finish a huge number of tasks at once (high throughput), even if each one is a little slower.

Analogy: A CPU is a sports car — gets one passenger across town fast. A GPU is a fleet of buses — each bus is slower, but together they move thousands of people. For adding two numbers, take the sports car. For adding a billion pairs of numbers, take the buses.

"But CPUs are multicore too!" — true, and they do real parallel work. A modern CPU has a handful of cores (≈8–64), often running two threads each (hyper-threading) — so tens of hardware threads. "A few fast workers" means few relative to a GPU, which keeps hundreds of thousands of threads in flight. So it's not one-vs-many — it's a few big, clever cores versus thousands of tiny, simple ones.

This shows up in the chip layout. CPUs spend silicon on caches and control logic to keep one thread sprinting. GPUs spend it on raw arithmetic units (ALUs):

By the numbers: An Nvidia A100 delivers ~19.5 TFLOPS at 32-bit precision; a 24-core Intel CPU around 0.66 TFLOPS (2021 figures). That's roughly a 30× throughput gap — and it keeps widening.

Concretely, in clock cycles: an add costs a handful of cycles; a read from main memory can cost hundreds. A CPU spends most of its transistors on caches and branch prediction so its one thread rarely has to wait that long. A GPU does the opposite — it keeps thousands of threads in flight, so the instant one stalls on memory, another is ready to run. It trades per-task speed for hardware that's never idle.

2 The architecture: SMs and cores

In plain words: a GPU is a bunch of mini-processors (called SMs) bolted together, each with its own little crew of cores.

A GPU is an array of Streaming Multiprocessors (SMs). Each SM packs many cores plus its own shared memory, schedulers, and special units (like tensor cores). The Nvidia H100 (SXM5) has 132 SMs, each with 128 FP32 cores — about 16,896 cores. (You'll sometimes see “8,448” quoted; that's counting the 64 FP64 cores per SM instead — same chip, different unit.)

Analogy: Think of the GPU as a factory. Each SM is a workshop with its own crew of workers (cores), a shared workbench (shared memory), and a foreman (scheduler). The factory runs many workshops in parallel.
GPU (factory)

Each SM is independent. More SMs = more work in parallel, which is why GPU generations keep adding them.

3 The memory hierarchy

In plain words: close memory is fast but tiny; far memory is huge but slow. Fast code keeps the data it's using close.

GPUs have layers of memory trading speed for size. Small + fast sits close to the cores; big + slow sits far away. Good GPU code keeps data in the fast layers as long as possible. (Hover a row.)

The key trick: The H100's global memory is huge (80 GB of HBM3) and has enormous bandwidth (~3.35 TB/s) but high latency — a single read can cost hundreds of cycles versus ~1 for a register. GPUs hide that delay by having thousands of threads ready, so there's always other work to run while some wait on memory.
Analogy: Registers are the tools in your hand. Shared memory is your workbench. Global memory is the warehouse across the yard — lots of stock, but a long walk. You batch your warehouse trips and keep what you need on the bench.

4 How code actually runs: grids, blocks, threads

In plain words: you write the work for ONE item, and the GPU runs thousands of copies — organized into a tidy threads-in-blocks-in-a-grid hierarchy.

You write a kernel — a small function (like "add element i of vector A to element i of vector B") that runs on the GPU. You then launch it across a small army of threads, organized in a hierarchy:

Thread — does the work for one data element.

Analogy: A grid is the whole job, a block is a team assigned to one workshop (SM), and a thread is a single worker.

The lifecycle of a kernel launch

  1. Copy in — move input data from CPU (host) memory to GPU (device) global memory.
  2. Schedule blocks — the GPU assigns blocks to SMs. All threads of a block stay on the same SM. Extra blocks wait in line for a slot.
  3. Execute in warps — each SM runs threads in groups of 32 (see below).
  4. Copy out — move results from GPU memory back to the CPU.

5 What the code actually looks like (CUDA)

In plain words: GPU code is just C with a few extra bits. You write one thread's job; the <<< >>> syntax fires off thousands of copies.

CUDA is Nvidia's way of writing GPU programs — basically C/C++ with a few extra keywords. A program splits in two: host code runs on the CPU and orchestrates everything; device code is the kernel that runs on the GPU. Here's the full vector-addition example the article mentions but skips.

Host code runs on the CPU
int n = 1 << 20;          // ~1M elements
size_t bytes = n * sizeof(float);

// CPU arrays h_A, h_B filled with data…
float *d_A, *d_B, *d_C;     // GPU pointers

// 1 · allocate memory ON the GPU
cudaMalloc(&d_A, bytes);
cudaMalloc(&d_B, bytes);
cudaMalloc(&d_C, bytes);

// 2 · copy inputs CPU → GPU
cudaMemcpy(d_A, h_A, bytes, cudaMemcpyHostToDevice);
cudaMemcpy(d_B, h_B, bytes, cudaMemcpyHostToDevice);

// 3 · launch the kernel as a GRID
int threads = 256;          // per block
int blocks  = (n + threads - 1) / threads;
vecAdd<<<blocks, threads>>>(d_A, d_B, d_C, n);

// 4 · copy result GPU → CPU
cudaMemcpy(h_C, d_C, bytes, cudaMemcpyDeviceToHost);

// 5 · free GPU memory
cudaFree(d_A); cudaFree(d_B); cudaFree(d_C);
Device code (the kernel) runs on the GPU
__global__ void vecAdd(const float *A,
                       const float *B,
                       float *C, int n)
{
    // which element is THIS thread responsible for?
    int i = blockIdx.x * blockDim.x
          + threadIdx.x;

    // guard: last block may overshoot n
    if (i < n)
        C[i] = A[i] + B[i];   // one add, that's it
}
  • __global__ marks a function that runs on the GPU but is called from the CPU.
  • <<<blocks, threads>>> is the launch configuration — it defines the grid (how many blocks, how many threads each).
  • The kernel is written from one thread's point of view. You don't write a loop over the array — you write what a single thread does, and the GPU runs it across thousands of threads.
The launch chevrons <<<blocks, threads>>> are the whole point: in one line you spawn blocks × threads copies of the kernel. With 1M elements and 256 threads/block, that's 4,096 blocks running the same 3-line function.

The one line everyone trips on

Every thread runs the same code, so how does each one work on a different array element? It computes a unique global index from its coordinates:

6 Warps: 32 threads in lockstep

In plain words: the GPU runs threads in packs of 32 that all perform the same step at the same moment.

Inside an SM, threads are executed in fixed groups of 32 called a warp. Every thread in a warp runs the same instruction at the same time, just on different data. This is called SIMT (Single Instruction, Multiple Threads).

Analogy: A warp is a rowing crew of 32. The coxswain calls one stroke and everyone pulls together — same action, different oar.

This connects back to blocks: the block size you chose in the code is just split into warps of 32. A 256-thread block becomes 8 warps; 1,024 threads becomes 32 warps. That's why block sizes are almost always multiples of 32 — anything else wastes lanes in the last warp.

Why high latency doesn't hurt: warp swapping

When a warp has to wait (e.g. for slow memory), the SM doesn't sit idle — it instantly switches to another warp that's ready to run. Because each thread keeps its own registers, switching costs nothing (unlike a CPU context switch). The compute units stay busy.

7 Warp divergence: when the 32 disagree

In plain words: if threads in a pack of 32 hit an if/else and don't all go the same way, the pack has to do both sides one after the other — so it takes twice as long.

Remember our kernel had an if (i < n). Here's the catch with running 32 threads in lockstep: they all share one instruction at a time. So what happens when a branch sends some threads one way and the rest another?

The hardware can't run two different instructions at once. So it runs the if threads first — the others sit idle — then runs the else threads while the first group sits idle. Both paths happen back-to-back. The warp takes as long as branch A plus branch B. This is warp divergence.

The takeaway: a branch only costs you when threads in the same warp split on it. Our if (i < n) is cheap — only the last warp has a few threads past the end. But something like if (data[i] > 0), where the answer varies thread-by-thread, can split every warp and roughly halve your speed. Good GPU code tries to keep neighbouring threads on the same path.

8 Memory coalescing: read together, read fast

In plain words: if 32 neighbouring threads read 32 neighbouring slots of memory, the GPU grabs them all in one trip. Spread those reads out, and it needs a separate trip for each.

Global memory isn't delivered one value at a time — it arrives in fixed-size chunks (think: a whole shelf, not a single item). When the 32 threads of a warp request 32 consecutive addresses, they all live on the same chunk, so the GPU fetches it in one transaction. That's coalesced access.

If those same 32 threads request scattered or strided addresses, each lands on a different chunk. Now the GPU does many transactions and throws away most of every chunk it fetched — same amount of data wanted, many times slower.

The rule of thumb: let thread i touch element i — exactly what our vector add does. Neighbouring threads → neighbouring memory → fewer, fuller transactions. This single habit is often the biggest difference between fast and slow real-world kernels.

9 Occupancy: keeping the SM full

In plain words: keep each mini-processor packed with threads, so there's always something ready to run while others wait on memory.

Occupancy is how full an SM is: the warps it's actually running divided by the most it could hold. The fuller it is, the more warps it can swap between to hide latency. But each SM has fixed budgets — registers, shared memory, and slots for threads and blocks. Your kernel's needs get carved out of those budgets, and whichever one runs out first caps how many threads fit. Ask for too much per thread, and fewer threads run at once.

Analogy: The SM is a parking garage with fixed spaces. If every car (thread) demands two spaces' worth of registers, only half as many park — half occupancy.

Try it: an H100 SM has 65,536 registers and room for 2,048 threads. Drag to see how registers-per-thread caps how many threads fit.

The tension: giving each thread more registers can make individual threads faster — but lowers occupancy, leaving the SM with fewer warps to hide latency. Tuning that trade-off is much of the art of GPU optimization.

10 Why AI runs on GPUs

In plain words: training AI is mostly giant multiplication tables — millions of tiny, independent sums. That's exactly the kind of work GPUs eat for breakfast.

Here's the payoff for everything above. Training and running a neural network is, underneath, mostly matrix multiplication. And a matrix multiply is built from dot products: every cell of the result is a row of one matrix dotted with a column of the other — and each result cell is completely independent of the others.

Sound familiar? That's the same "independent iterations" idea from the very first section. A 1000×1000 output is a million independent dot products → a million threads. Click any output cell to see the one row and column that produce it:

On top of this, modern GPUs add tensor cores: dedicated units that multiply a small matrix in a single operation — precisely the inner step of deep learning. So the traits that make a GPU great at adding two vectors (mountains of independent arithmetic, latency hidden behind thousands of threads) are exactly what make it the engine of modern AI.

Full circle: "GPU for AI" isn't magic — it's that neural networks are enormous piles of independent multiply-adds, and a GPU is a machine built to do enormous piles of independent arithmetic.

11 Beyond GPUs: TPUs & other accelerators

In plain words: GPUs aren't the only AI chips. TPUs and friends specialize even harder for matrix multiply — trading flexibility for more speed and efficiency.

Once you know that AI is mostly matmul (§10), every AI chip makes sense as a different bet on how hard to specialize for matmul. Picture a spectrum from general-purpose to single-minded — (click each):

A GPU's tensor cores are already a step down this road — little matmul engines bolted inside a general chip. A TPU is what you get if you make the whole chip that: it throws out general cores and hardwires a giant systolic array (a physical grid of multiply-accumulate units, 128×128 or 256×256, that data flows through). Less flexible than a GPU, but more matmul-per-watt. TPUs are Google's (rent them on Google Cloud; latest are v6 "Trillium" and v7 "Ironwood").

✓ The whole picture, in one breath

A GPU is a factory of SMs (workshops), each packed with cores. You write a kernel — the instructions for one worker — and launch a grid of threads with <<<blocks, threads>>>. Each thread computes its own global index (blockIdx·blockDim + threadIdx) to find the data it owns. Threads are grouped into blocks assigned to single SMs, then run 32-at-a-time as warps, all executing the same instruction on different data (SIMT). Memory is layered from instant registers up to a huge-but-distant global memory, and the GPU hides that distance by instantly swapping to another ready warp whenever one waits — for free, because every thread keeps its own registers. How full you keep each SM with warps is occupancy, and balancing it against register and memory limits is most of GPU performance work.
