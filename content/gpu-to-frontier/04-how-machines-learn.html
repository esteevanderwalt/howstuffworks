How machines learn: gradients & autograd

A model is millions of numbers. "Learning" is just nudging each one to be a little less wrong — over and over. Here's the machinery that decides which way to nudge.

▶ Start here: turning knobs to reduce error

In plain words: a model is a giant box of number "knobs." Learning = measure how wrong it is, then turn every knob slightly toward "less wrong." Repeat millions of times.

Explainer #3 left us with a question: stacking matmuls computes things — but how does a model learn the numbers inside those matrices? Here's the whole idea in one picture.

Imagine a single knob and an error meter. Turn the knob a touch; the error moves. If it went down, you turned the right way; if up, the wrong way. Now scale that up: a real model has millions of knobs (its weights), and one error meter (the loss). Learning is turning all the knobs, together, toward lower error.

With millions of knobs, how do you know which way to turn each one — without trying them one at a time?

The answer: the gradient tells you, for every knob at once, which way and how hard to turn it. Computing it efficiently for millions of knobs is backpropagation, and the system that does it automatically is autograd. That's this whole explainer.

1 Loss: one number for "how wrong"

In plain words: the loss is a single number that measures how far the model's predictions are from the right answers. Smaller is better; learning means making it small.

Before you can improve, you need a score. The loss function boils the model's wrongness down to one number. A common one is squared error — for each example, take (prediction − target), square it, average over the batch:

Loss mean squared error
loss = mean((prediction - target)**2)

Everything the model does — every matmul from #1–3 — feeds into this one number. Learning is the search for weights that make the loss small. So the real question is: which way do we move the weights?

2 The gradient: which way is downhill

In plain words: the gradient is the slope of the loss with respect to each weight — nudge the weight a hair, and the gradient says how much the loss moves, and in which direction.

(Calculus refresher, no jargon:) a derivative answers one question — "if I nudge this input a tiny bit, how much does the output change, and which way?" That's it. A big positive derivative means "push this up and the loss shoots up" (so push it down); near zero means "this knob barely matters right now."

The gradient is just all those derivatives at once — one slope per weight, bundled into a vector. It points in the direction the loss increases fastest. So to reduce the loss, you step the opposite way. That single rule — step against the gradient — is called gradient descent.

Analogy: you're on a foggy hillside (the loss surface) and want the valley. You can't see far, but you can feel the slope under your feet. The gradient is that slope; you take a step downhill, then feel again, and repeat.

3 Gradient descent & the learning rate

In plain words: take a downhill step, re-measure, step again. The learning rate is how big each step is — too small crawls, too big overshoots and blows up.

Each step is dead simple: weight = weight − learning_rate × gradient. The learning rate is the one knob you tune by hand. Drag it and watch the ball roll down the loss curve — small steps creep, a good rate converges fast, and too-big steps bounce out of the valley entirely:

That's the entire learning algorithm: step downhill, repeat. Everything else is about computing the gradient — which, for a deep model, is the clever part.

4 The chain rule: gradients through a graph

In plain words: a model is a chain of simple operations. To find how an early weight affects the final loss, you multiply the little local effects along the chain. That's the chain rule.

A model isn't one operation — it's a computational graph of simple steps (multiply, add, activate) feeding into each other, ending at the loss. Each step knows its own local derivative ("if my input wiggles, my output wiggles this much"). The chain rule says: to get the effect of a weight deep in the graph on the final loss, multiply the local derivatives along the path between them.

So gradients are computed in two sweeps:

  • Forward pass — run the graph left-to-right, computing every value, ending at the loss.
  • Backward pass — start at the loss with gradient 1, and walk right-to-left, multiplying local derivatives to hand each node its gradient. This is backpropagation.

Take a tiny graph — d = a·b, then L = d + c:

a b → × → d c → + → L (loss)
① Forward left → right, build the loss
a, b, c        # inputs / weights
d = a * b      # multiply
L = d + c      # add → the loss
② Backward loss → left, chain rule
dL/dL = 1                # seed
dL/dd = 1                # L = d + c
dL/dc = 1
dL/da = dL/dd · b        # d = a · b
dL/db = dL/dd · a

The key efficiency: backprop computes the gradient for every weight in a single backward sweep — not by re-running the model once per weight. That's what makes training millions of parameters feasible.

5 Build a tiny autograd

In plain words: you don't do the chain rule by hand. Each value remembers how it was made and a tiny rule for pushing gradient back — string them together and backprop runs itself.

This is the trick at the heart of every deep-learning framework, and it's small. Make each number a little object — a Value — that records its parents (what it was computed from) and a local backward rule. Then one traversal does the whole chain rule:

micrograd-style autograd the whole idea
class Value:
    def __init__(self, data, parents=()):
        self.data = data
        self.grad = 0            # dLoss/dself, filled in by backward
        self._parents = parents
        self._backward = lambda: None   # local rule

    def __mul__(self, other):
        out = Value(self.data * other.data, (self, other))
        def _backward():        # chain rule for a*b
            self.grad  += other.data * out.grad
            other.grad += self.data  * out.grad
        out._backward = _backward
        return out

Call loss.backward() and it seeds grad = 1 at the loss, then walks the parents in reverse, running each node's _backward — exactly the right-to-left sweep from §4. Every weight ends up with its gradient, automatically.

The punchline (Karpathy): "These ~94 lines are everything needed to train a neural network. Everything else is just efficiency." PyTorch (explainer #5) is this same idea, scaled up and running its operations as GPU matmuls.

6 The training loop

In plain words: learning is four steps in a loop — predict, score, find the gradients, nudge the weights. Run it thousands of times and the model gets good.

Put it all together and the entire act of "training" is this little loop:

The training loop PyTorch-flavoured
for batch in data:
    pred = model(batch.x)              # 1. forward
    loss = loss_fn(pred, batch.y)      # 2. score
    loss.backward()                    # 3. gradients (backprop)
    optimizer.step()                   # 4. nudge weights downhill
    optimizer.zero_grad()              #    reset for next round

Forward → loss → backward → step. That's it. The optimizer is just gradient descent (§3) wrapped up — step() applies weight −= lr × grad for every weight at once. Repeat over the data many times (epochs) and the loss falls.

7 Backward is also matmul

In plain words: the backward pass isn't a different kind of work — it's more matrix multiplies. So training runs on the exact GPU machinery from explainers #1–3.

A layer's forward pass is a matmul: y = W · x. Its backward pass — computing the gradient w.r.t. the input and the weights — is also matmuls (with transposed versions of W and x). So a training step is just two passes of matmul: one forward, one (roughly twice as much work) backward.

Why this matters: everything you learned about making matmul fast — tiling, tensor cores, fp16/bf16 (#3) — applies directly to training, not just inference. Learning is GPU-bound for the same reason inference is: it's matmul all the way down.

8 Why deep networks are hard to train

In plain words: backprop multiplies many local derivatives together. If they're mostly under 1, the gradient shrinks to nothing by the early layers; if over 1, it explodes.

The chain rule's multiplication is also its weakness. In a deep stack, the gradient reaching an early layer is a product of many local derivatives. Two failure modes:

  • Vanishing gradients — classic activations like sigmoid have derivatives ≤ 0.25. Multiply dozens of sub-1 numbers and the product collapses toward zero, so early layers barely learn.
  • Exploding gradients — if local derivatives are above 1, the product blows up, and updates become wild and unstable.

The fixes are the scaffolding of modern deep learning: ReLU activations (derivative of 1, no shrinkage), residual connections (gradient shortcuts), normalization (batch/layer norm), and gradient clipping (cap the size). They're why we can train very deep models at all — and several reappear inside the transformer (#7).

✓ The whole picture

Everything you just read, as one chain — from a loss to a trained weight:

loss→gradient→chain rule→backprop / autograd→loop: fwd→loss→back→step
